The data my app generates or uses is situated on two databases; an SQLLite database which is built into the Android app and the MySQL database on the server.

The MySQL database currently has two tables - favouritePlaques and Users. In the future, I will have a visitedPlaques table (same structure as favouritePlaques) and a table for routes/collections. This will work by giving each Plaque in the same collection a collection_id and then a collections table will define the name and other details of that route, and the Primary Key will be a foreign key to the collection_id.

  • Right now there's only 1 user, because only I have registered. In the future this table will probably have a lot more fields, including password and gender).

  • This is the table for which plaques have been favourited by which users. Right now they're all by me, because I'm the only user (!). As I said on the Server Side page, a Unique Key exists between the plaque_id and the user_id to ensure that the combination of those two is unique before being inserted.

  • This is the SQLLite database, which holds 2609 rows of plaque data, sourced from two JSON files (read more in the Technical Report). I need to add a few fields to this - a points column, for each plaque and a collection_id column. Also, there is an area column (currently they all show London), which could be deleted - however, if I decided to add other locations into PlaqueIt then I may need that column so may keep it in.

I have also developed 3 data models to support the development; an Entity Relationship model, based on the notation developed by Peter Chen in 1976 .
